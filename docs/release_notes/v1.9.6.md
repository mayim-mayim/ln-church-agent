# Release v1.9.6 — Normalized Unmapped Observation Submission

This release introduces a dedicated, public-safe path for submitting observations of unmapped, unknown, or unsupported payment surfaces to the LN Church Observatory, ensuring strict separation from actual verified payment proofs.

## Added
* **Normalized Unmapped Submission**: Added explicit `submit_unmapped_observation()` and `submit_unmapped_observation_async()` methods to `LnChurchClient`.
* **Standardized Normalization**: Automatically normalizes `payment_scheme_unmapped`, `unsupported_challenge_shape`, and `unknown_rail` into formal, public-safe External Observation payloads.
* **Secret Stripping**: Enforces rigorous redaction (`_strip_secrets_from_evidence`) on any `extra_protocol` parameters passed to unmapped observations.

## Maintained Constraints
* **Conservative Auto-Submit**: The default `ELIGIBLE_CODES` for auto-submission remains strictly conservative. Explicit opt-in (e.g., `--include-unmapped`) is required to submit unmapped observation signals.
* **Navigation Guardrails Unchanged**: The `execute_request()` unsafe HATEOAS / cross-origin navigation guardrails (`NavigationGuardrailError`) remain fully intact.
* **No Execution Changes**: No payment execution paths or local policy evaluations were altered.

## Clarification
* Unmapped observations are treated strictly as **discovery signals**, not payment proofs. The SDK forces `payment_performed=False` and `verification_status="unverified"` on all payloads routed through these new methods.
* Developers should use these methods rather than manually attempting to POST raw data to the observation endpoint via `execute_request()`.
