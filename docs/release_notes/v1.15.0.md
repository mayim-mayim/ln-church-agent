# Release v1.15.0 — Verified Domain Sponsor (MVP v1)

This release expands the Paid Domain Observation functionality by introducing the **Verified Domain Sponsor** layer to the SDK and CLI.

## Added
* **Sponsor Challenge API**: `create_domain_sponsor_challenge()` to issue a deterministic challenge payload.
* **Sponsor Verify API**: `verify_domain_sponsor()` to request the LN Church observatory to fetch and verify the challenge document from the target domain.
* **Local Save Helper**: `save_domain_sponsor_challenge_document()` safely extracts and saves the raw JSON challenge document to `.well-known/ln-church-domain-sponsor.json` without leaking request payloads.
* **CLI Expansion**: Added `observe-domain sponsor challenge` and `observe-domain sponsor verify` subcommands.

## 🛡️ Safety Boundaries & Strict Non-Goals

The Verified Domain Sponsor feature strictly asserts one fact:
> *The sponsor was able to publish a specific challenge document to the target domain's public web surface.*

It explicitly **DOES NOT** mean any of the following:
* **Not Legal Ownership Proof**: This is domain-control only.
* **Not a Trust Score**: Verified domains are not ranked higher or trusted inherently.
* **Not an Endorsement / Certification**: The LN Church does not endorse the target site.
* **Not a Security Scan**: The domain is not guaranteed to be safe from vulnerabilities.

## 🔒 Secret Redaction & Anti-Automation

* **No Automatic Execution**: The SDK intentionally **does not** automatically issue challenges after a paid observation slot is registered, nor does it automatically poll for verification.
* **Secret Leakage Prevention**: The `challenge_token`, `X-LN-Result-Handle`, `X-LN-Request-Hash`, and `X-Internal-Secret` are securely handled in memory. The CLI explicitly suppresses these secrets from standard output to prevent log poisoning.
* **No SDK-Side Fetching**: The SDK never directly fetches or validates the challenge file against the target domain. Verification is exclusively routed through the LN Church backend API to prevent SSRF vulnerabilities and maintain centralized integrity.
