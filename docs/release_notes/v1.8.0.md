# Release v1.8.0 — Agent Commerce Surface Inspector & APP-aware Detection

This release expands the CLI `inspect` tool to act as a generalized **Agent Commerce Surface Inspector**. 

It safely detects higher-order commerce protocols—starting with OKX Agent Payments Protocol (APP)—without attempting to execute the payment logic.

## Added
* **Commerce Layer Detection**: The SDK now detects OKX APP metadata (`protocol: okx-app`, `broker` objects, intents like `escrow`/`upto`/`batch`) via the new `detect_app_surface` heuristic engine.
* **Extensible Commerce Fields**: Added `commerce_protocol`, `commerce_intent`, `commerce_transport`, and `settlement_method` to `InspectResult`. *These fields are explicitly designed to accommodate future commerce standards such as Google AP2, ACP, or UCP as they mature.*
* **Rail Co-existence**: The SDK now separates Commerce layer detection from Settlement rail parsing. A challenge can now return `rails_detected: ["APP", "x402"]` if an APP layer wraps an x402 settlement payload.
* **Strict Heuristics**: Prevents false positives. A standard `x402 exact` challenge on X Layer (`eip155:196`) will not be misclassified as APP without explicit protocol or broker metadata.

## Not Changed (Intentional Constraints)
* **No APP payment execution**: This release does **not** execute OKX APP. It defaults to `observe_only` (or `stop_safely` for `session`/`escrow` intents).
* **No AP2 / ACP detection**: While the fields are prepared, detection for Google AP2/ACP is not implemented in this patch.
* **No wallet/signer changes**: Existing `L402`, `x402`, and `MPP` execution routines remain exactly as they were.
