# Release v1.9.1 — Guided Handoff for Agent Commerce Surfaces

Following the detection capabilities introduced in v1.9.0, this release adds **Guided Handoff** to the SDK's inspect layer. 

The goal of this update is to prevent AI agents from "freezing" when they encounter complex commerce protocols (like Google AP2, ACP, or OKX APP). Instead of simply throwing an `observe_only` or `stop_safely` status, the SDK now outputs structured, deterministic guidance indicating exactly *what information to ask for* and *what actions to avoid*.

## 🎯 The "Guided Handoff" Principle
The SDK is a buyer-side settlement runtime. It intentionally **does not** execute higher-order commerce negotiations (like signing AP2 mandates or completing ACP checkouts). 

Instead, it safely hands off the transaction to the AI's Operator Approval flow, providing the LLM with the context it needs to make safe decisions.

## ✨ Added Fields to `InspectResult`
When a commerce surface is detected, `InspectResult` now populates the following fields:
* `handoff_mode`: Set to `"guided_handoff"`.
* `approval_required`: Set to `True`.
* `operator_approval_reason`: Provides the specific context (e.g., `"commerce_surface_with_settlement_rail"`, `"malformed_or_unsupported_settlement_hint"`).
* `ask_site_for`: A list of required clarifications (e.g., `["quote_details", "mandate_scope", "settlement_rail_options"]`).
* `do_not`: A list of strict prohibitions (e.g., `["treat_mandate_as_settlement_proof"]`).
* `required_evidence`: The minimum data required before the operator should approve the transaction (e.g., `["explicit_price", "merchant_identity"]`).
* `missing_information`: A heuristically determined list of required evidence that is visibly missing from the current payload.

## 🛡️ Raw Secret Redaction
To maintain high security, the SDK strictly prevents raw secrets from leaking into the `InspectResult` JSON output.
While the SDK correctly identifies the `authorization_artifact` (e.g., `"shared_payment_token"`), the actual secret values (mandates, broker tokens) are safely stripped from the serialized output.

## 🔄 Maintained Behaviors
* **Standard Paths Unchanged**: Normal `L402`, `x402`, and `MPP` paths do not trigger a guided handoff and continue to return `pay_and_verify`.
* **Co-existence Safety**: If a valid settlement rail (like x402) co-exists with a commerce surface (like AP2), the SDK maintains its `observe_only` guardrail and will **not** auto-escalate to payment execution.
