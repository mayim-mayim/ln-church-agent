# Release v1.16.2 — Buyer-side Payment Boundary Hardening

This candidate hardens buyer-side HTTP 402 execution while retaining the v1.16.1 public API and serialization fields.

## Canonical payment requirements

Exact x402 requirements are normalized once from the selected `accepts` entry. Policy evaluation, EVM/SVM signer input, output verification, and ambiguous-spend reservation use that canonical atomic amount. Missing canonical requirements, non-canonical integer amounts, unknown network/token pairs, malformed destinations, and contradictory outer or nested values fail before signer invocation.

## Lightning and MPP

`bolt11>=2.1.0,<2.1.1` is a mandatory dependency. Version 2.1.0 uses the Windows-wheel-compatible `coincurve` backend, so standard Windows installs do not build native `secp256k1`; the fail-closed BOLT11 signature, checksum, and amount validation boundary is unchanged. L402, Payment, and MPP invoices are decoded and validated before wallet or delegate invocation. Invalid checksums, amountless invoices, placeholders, missing macaroons, and declared amount/currency mismatches fail closed. MPP auth-parameter and JSON semantic field names are case-insensitive; duplicate aliases, empty or falsey declarations, non-canonical amounts, and contradictions between flat, top-level, and `methodDetails` representations are rejected. A `Payment request` must be a complete and internally consistent draft before the legacy fallback setting is considered. Partial, invalid, unsupported-method, or contradictory method/intent declarations fail before wallet, delegate, signer, HTTP retry, Evidence-success, or budget effects regardless of that setting. A complete draft retains the existing explicit opt-in fallback behavior. MPP declarations accept `SAT` and `SATS`, normalize both to `SATS`, and use the Invoice value only when amount/currency are genuinely absent. There is no regex amount fallback.

## EVM and SVM signer boundaries

EIP-3009 output is checked against trusted chain/contract metadata, authorization fields, time bounds, signature format, and the recovered signer. EVM/SVM credential generation and local validation complete before the operation is marked irreversible, so signer exceptions and invalid signer output remain non-ambiguous and do not reserve session spend. The v1.16.1 `EVMSigner` Protocol member set remains unchanged; atomic generation is detected as an optional concrete capability.

SVM exact output is decoded as a versioned transaction. Compute Budget instructions are accepted only when absent or when exactly one `SetComputeUnitLimit` and one `SetComputeUnitPrice` are present. The internal bounds are limit `<= 200,000` CU, price `<= 1,000,000` micro-lamports/CU, and ceiling total priority fee `<= 200,000` lamports; the built-in adapter's `200,000`/`1` pair is accepted. The validator also permits the bounded memo form plus one SPL Token `TransferChecked`, and verifies the canonical network/mint, source and destination token accounts, atomic amount, fee payer, authority, and source-owner signature.

## Navigation and operation state

Redirect and HATEOAS destinations are checked against local policy before sending. Policy authority matching is case-insensitive but keeps host-only, explicit `:443`, and alternate explicit ports distinct. Origin comparison separately uses scheme, normalized hostname, and effective port. Credential-bearing header families and recursively nested payment secrets are normalized across camelCase, snake_case, hyphen, and whitespace spellings before removal. Cross-origin navigation discards all original payload data and sends only recursively sanitized suggested data where applicable. Evidence submissions use the same credential-family treatment while retaining public hashes, references, fingerprints, presence flags, authorization scheme, and token asset identifiers.

One internal operation fingerprint, including the top-level caller-supplied Idempotency Key when present, is carried across sync and async paid retries and HATEOAS recursion. A different explicit key can identify a new top-level purchase in the same context. Completed, cached-credential, and ambiguous states stop another irreversible attempt. If an irreversible adapter call loses its result, the canonical amount is reserved once for the operation; validation failures before that call do not reserve spend. Pre-irreversible transport and validation failures retain their compatible public exception type while exposing only a fixed secret-free message. Evidence import/export is best-effort: repository failure cannot replace a primary sanitized exception, turn a successful paid retry into a failure, or add repository secrets to the public exception chain.

## Compatibility and dependencies

The version remains `1.16.2`. Public constructors, public execution method signatures, all six `ExecutionContext` fields and their types/defaults, existing Protocol required members, and public exception direct bases match v1.16.1.

The package declares Python `>=3.8.1`, matching mandatory `bolt11>=2.1.0,<2.1.1`, and installs `httpx` without the SOCKS extra. CLI inspection annotations use Python 3.8-evaluable `typing` forms. The clean validation environment used Python `3.12.13` and resolved `bolt11 2.1.0`, `httpx 0.28.1`, `solana 0.39.0`, `solders 0.27.1`, and `x402 2.15.0`; this does not claim that every allowed optional dependency version was tested. A Python 3.8.x runtime was not available in that environment.
