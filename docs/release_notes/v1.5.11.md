# Release v1.5.11 — Interop Matrix Separation & Live Diagnostics

This release focuses on **Observability** and **Resilience**. Moving beyond the deterministic LN Church Sandbox, version 1.5.11 empowers agents to navigate and debug the unpredictable nature of the live Lightning Network.

## Key Changes

### 1. Intentional Mismatch Separation
You can now mark runs in the Sandbox Harness as "Validation Tests" to ensure that purposeful protocol deviations do not pollute your production quality metrics.
* **New Fields**: `comparison_class` (`production_like` | `validation_test`) and `test_mode` (`normal` | `intentional_mismatch`).
* **UI**: The Interop Matrix now displays a purple "INTENTIONAL" badge for these runs.

### 2. Precise Executor Attribution
We fixed a critical reporting bug where failed delegated executions were incorrectly attributed to the Native engine. Attribution is now locked in before the request lifecycle begins, ensuring accurate audit trails even in failure states.

### 3. Heuristic Live Diagnostics
To handle "the wild west" of external APIs, the SDK now analyzes error patterns (like Cloudflare 5xx codes) to pinpoint the failure origin.
* **Suspected Origin**: Heuristically determines if the crash happened at the **Target API** or within your own **Payment Backend** (e.g., LNBits timeout).
* **HTML Metadata Extraction**: Automatically scrapes failing hostnames from infrastructure error pages to assist in rapid troubleshooting.

### 4. Optimized Settlement Buffers
Real-world Lightning routing often exceeds standard HTTP timeout expectations. We have increased the LNBits status-check delay to 5 seconds, significantly improving the success rate of state synchronization after a successful payment.

## Developer Note
**Delegated Mode is currently Experimental.** While verified against the LN Church Harness, broad compatibility with all L402 providers is not yet guaranteed. When encountering `Protocol Success: False` on live targets, always run with the `--debug` flag. Check if `suspected_failure_origin` points to `PAYMENT_BACKEND`.
