# Release v1.9.0 — AP2 / ACP Inspect-Only Commerce Surface Detection

This release expands the Agent Commerce Surface Inspector to safely detect and classify emerging agentic commerce protocols such as Google AP2 (Agent Payments Protocol) and ACP (Agentic Commerce Protocol). 

## 🔍 Architecture Principles
* **Inspect-Only:** AP2 and ACP are treated strictly as commerce, authorization, or checkout surfaces. They are **not** settlement rails.
* **No Execution:** The SDK does **not** execute AP2/ACP mandates, checkout completions, or payment token delegations. Authorization metadata is not treated as a settlement proof.
* **Observe & Halt:** Detection of these surfaces defaults to `observe_only`. Even if a valid HTTP 402 settlement challenge co-exists with AP2/ACP metadata, the SDK prioritizes the commerce surface and suppresses the `pay_and_verify` recommendation to prevent unauthorized mandates.
* **Malformed Overlap Protection:** If an AP2/ACP surface co-exists with an unsupported or malformed settlement hint, the SDK will explicitly return `stop_safely` to protect the agent.

## ✨ Added
* **Commerce vs Rail Separation:** The `InspectResult` model now explicitly separates `surfaces_detected` (e.g., `["AP2", "ACP"]`) from `settlement_rails_detected` (e.g., `["x402", "L402", "MPP"]`).
* **Granular Surface Metadata:** Added `surface_type`, `detection_confidence`, `detection_reason`, and `unsupported_reason` fields to provide rich context to the observing LLM.
* **Comprehensive Detection:** Heuristically classifies AP2 `payment_mandate` / `checkout_mandate` and ACP `agentic_checkout` / `delegated_payment` from HTTP 402 payloads.

## 🛡️ Maintained Constraints
* **Existing Settlement Paths Unchanged:** 100% backward compatibility for existing L402, x402, and MPP execution behaviors.
* **Legacy APIs Intact:** The `rails_detected` and OKX APP metadata fields from v1.8.0 are fully maintained.
* **Sandbox & Evidence Intact:** Sandbox Evidence Corpus Candidates and Grant Overrides remain unaffected.
