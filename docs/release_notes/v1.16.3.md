# Release v1.16.3 — P0-2 Canonical Payment and Privacy Boundaries

Version 1.16.3 packages the audited P0-2 Agent changes accumulated after the Public `v1.16.2` source tag. This Private release-preparation branch does not itself perform Public promotion, create a tag, or upload to PyPI.

## Canonical payment boundary

The high-level sync and async payment paths preserve one canonical requirement from the selected challenge through policy approval, credential generation, retry, receipt construction, and evidence restoration. The canonical record binds the final wire URL (including GET query parameters), HTTP method, request identity, caller idempotency key, expiry, and requirement hash. A challenge resource URL that differs in scheme, host, port, path, or query fails before payment. Immediately before signing and sending, the client rechecks the approved snapshot and rejects any mutation.

Payment, proof, and receipt identities incorporate the request identity, idempotency key, and canonical requirement hash so distinct logical operations do not collide or become incorrectly deduplicated when evidence is restored.

## Settlement and session-budget state

Credential generation is distinct from settlement confirmation. A request whose post-send result cannot be determined enters `settlement_unknown` instead of being treated as paid; external confirmation can recover it to `confirmed_not_paid`. Signature verification and settlement verification remain independent, and the absence of a settlement checker leaves `settlement_verified=false`.

Session-budget checking and reservation are performed atomically under the shared policy lock. Successful settlement confirms the reservation, while confirmed non-payment or pre-settlement failure releases it. Operation journals prevent evidence replay from double-counting a purchase.

## Canonical SVM exact boundary

Canonical SVM exact high-level sync and async auto-payment remains intentionally fail-closed because recent-blockhash validity is expressed in block height and cannot be proven to expire at or before the canonical Unix `expires_at`. The high-level lane stops after the approved canonical snapshot is fixed and before session-budget reservation or release, private-key or payment-credential inspection, fee-payer access, signer invocation, Solana RPC, credential generation, or a second HTTP request. The lower `_process_payment` guard remains as defense in depth.

The low-level SVM transaction builder and validator remain available for payload construction and verification only; they do not make high-level canonical auto-payment executable. The builder emits:

1. `SetComputeUnitLimit`
2. `SetComputeUnitPrice`
3. `TransferChecked`
4. Exactly one Memo

Without `extra.memo`, the Memo is 16 random bytes encoded as 32 lowercase hexadecimal characters. With `extra.memo`, it is the specified UTF-8 value. Memo values longer than 256 UTF-8 bytes are rejected. Both Memo variants were accepted by the x402 Python 2.16.0 facilitator verifier in the independent interoperability check.

## Privacy, receipts, and redirects

Evidence, receipt, error, navigation, and reporting boundaries remove raw macaroons, preimages, Authorization values, payment credentials, receipt tokens, signatures, private keys, sensitive nested fields, and URL-query values while retaining public hashes, fingerprints, references, and presence flags. Receipt signature verification does not imply settlement verification, and unsigned receipts remain unsigned.

Redirect processing strips credential-bearing header families case-insensitively. Crossing an origin or trust boundary also removes `Signature`, `Signature-Input`, and `DPoP`; HTTPS downgrade and unapproved destinations remain blocked.
