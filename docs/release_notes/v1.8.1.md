# Release v1.8.1 — Inspector Robustness & OpenClaw Stability

This hotfix release focuses on improving the robustness of the Inspector tool for scouting external 402 endpoints and enhancing its integration efficiency within discovery workers such as OpenClaw.

## Changes

### 1. Inspector CLI & Worker Hardening
* **Python API Recommendation**: Developers can now import and use `ln_church_agent.cli.inspect_url` directly. This enables stable execution from workers without relying on system `PATH` configurations.
* **Improved Subprocess Invocations**: Formally supported execution via `python -m ln_church_agent.cli inspect`, ensuring the tool runs within the intended virtual environment.

### 2. Expanded x402 Challenge Detection
The parser has been strengthened to automatically detect a wider variety of challenge shapes from both headers and bodies:
* **Alchemy-style**: Challenges with the `accepts` array located directly in the JSON body.
* **Custom Roots**: Root JSON objects containing `x402Version`, `paymentRequirements`, or `resource`.
* **Authorization Header**: Support for `X402` as a scheme within the `WWW-Authenticate` header.

### 3. Automated Rail Classification for Payment Scheme
The SDK now maps the IETF `Payment` scheme to practical settlement rails instead of treating it merely as a generic authorization method:
* `Payment` + `method: lightning/invoice` → `rails_detected: ["Payment", "MPP"]`
* `Payment` + `method: eip3009/exact/evm` → `rails_detected: ["Payment", "x402"]`

### 4. Advanced Diagnostic Output
`InspectResult` now includes granular information to help downstream AI agents or systems determine the specific cause of a parsing failure:
* **`diagnostic_class`**: Includes identifiers such as `x402_parse_error`, `unsupported_challenge_shape`, or `invalid_payment_auth_request`.
* **`error_stage`**: Identifies the specific phase of failure, such as `parse`.

## Fixed Issues
* **Numeric Protection**: Prevented crashes when the `amount` field contains invalid strings or objects by introducing a `_safe_float` utility that defaults to `0.0`.
* **Graceful Unknown Handling**: When an unknown scheme (e.g., `UnknownScheme`) is detected, the SDK now safely reports it as `reject_invalid` with `ok: True` instead of crashing, preserving existing test expectations.
