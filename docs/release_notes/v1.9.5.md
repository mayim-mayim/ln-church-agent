# Release v1.9.5 — Settlement Options Observation Layer

This release significantly enhances the SDK's capacity as a "Paid Surface Observer" without altering any core payment execution loops. It enriches inspection capabilities to expose the entire spectrum of settlement requirements an endpoint provides, turning the agent into a high-fidelity observation probe for the HTTP 402 ecosystem.

## Added
* **SettlementOption Model**: Introduced a safe, structured representation for tracking individual payment requirements extracted from x402 `accepts[]` arrays.
* **Full Options Enumeration**: The `inspect` tool and MCP payload now emit the complete `settlement_options` list alongside the explicitly `selected_settlement_option`. It distinguishes between what is merely presented and what the SDK's local policy would actually choose.
* **Execution Support Labeling**: Options are explicitly tagged with `execution_support` (`supported`, `supported_but_not_executed_in_inspect`, `observe_only`, `unsupported`, `unknown`) to separate observable settlement requirements from inspect-time execution behavior. Note that this denotes handling within the *inspect context*; for example, `scheme: "exact"` is treated as `observe_only` during external observation or post-settlement diagnostics, regardless of underlying SDK execution path availability.
* **Missing Information Augmentation**: For AP2/ACP/APP commerce surfaces lacking concrete 402 settlement hints, the inspector autonomously appends `settlement_rail_not_declared`, `network_not_declared`, etc., to the `missing_information` guidance.

## Improved
* **MCP Payload Preservation**: The `build_mcp_observation_payload` tool now dynamically inherits `network` and `asset` from the `selected_settlement_option` or the fallback options list. This prevents critical protocol metadata from dropping to `unknown` during pure observation flows. Additionally, `settlement_options_summary` has been elevated to the top level of the MCP observation payload for improved downstream ingestion.
* **LN Church Observatory Metadata**: Local inspect and observe outputs now feature a clear, non-intrusive tip explicitly bounding the observation as local, confirming that transmission to the LN Church Observatory is strictly an opt-in action.

## Maintained Constraints (No Breaking Changes)
* **Zero Execution Path Alteration**: The addition of full `accepts[]` enumeration strictly operates in observation scope. The actual payment loop (`execute_detailed`) and parser selection logic remains 100% unaltered.
* **No Auto-Submission**: The SDK fundamentally preserves its privacy-first posture. Observations and telemetry are **never** automatically sent to the Observatory.
* **Strict Inspect-Only for AP2/ACP**: Agent commerce surfaces remain non-executing.
