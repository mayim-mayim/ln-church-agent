# Release v1.12.0 — Reporter Identity Verification Layer

v1.12.0 introduces the **Verifiable Reporter Identity Layer**, enabling external agents to provide cryptographic proof of key control when submitting telemetry. This layer is strictly an identity verification tool, not a report validation or global trust scoring engine.

## Added
* **Identity Challenge Endpoint (`GET /api/agent/identity/challenge`)**: Generates deterministic, time-bound nonce challenges associated with a normalized `agentId` (currently supporting EVM keys, with Solana, Nostr, and LN pubkeys deferred as future scope).
* **Identity Verify Endpoint (`POST /api/agent/identity/verify`)**: Validates the cryptographic signature against the active challenge and records a `key_control_verified` status inside `AgentProfiles` valid for 7 days.
* **SDK Identity Methods**: Added `LnChurchClient.ensure_reporter_verification()` and its async counterpart `ensure_reporter_verification_async()` to automatically fetch, sign, and verify the identity challenge, caching the verified status in-memory.

## Safety Boundaries & Non-Goals
* **Optional Layer**: Identity verification is strictly optional. Runtimes and registries **must not** reject `self_reported` telemetry or treat unverified reporters as inherently unsafe or malicious.
* **Key-Control Verification Only**: This layer proves only that the reporting agent holds the private key corresponding to its declared `agentId`. It **does not prove report truth**, correctness, or achieve consensus on the validity of submitted observations.
* **No Automatic Invocations**: To prevent unexpected token or budget drainage, the SDK **does not automatic-call** verification hooks from `execute_detailed()` or `submit_goal_attempt_observation()`. Verification must be requested explicitly via `ensure_reporter_verification()`.
* **Terminology Restraints**: This layer explicitly avoids vocabulary like "trusted," "certified," "reliable," or "Virtue" assignments. It is not an economic payment proof.

## Supported Telemetry Contexts
The resulting reporter verification metadata is seamlessly attached and carried through downstream aggregation layers:
1. **Goal Attempt Memory** (`Lambda_GoalAttemptIngest.js`)
2. **External Observations** (`Lambda_ExternalObserve.js`)
3. **Failure Observations** (`Lambda_FailureObserve.js`)

This enables future read models to differentiate between anonymous `self_reported` claims and verifiable `key_control_verified` identity groups.
