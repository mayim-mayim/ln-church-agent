# Release v1.5.0 — Source-Agnostic Trust & Provider-Agnostic Outcome

This release enhances the "Decide & Verify" architecture introduced in v1.4.0 by deeply abstracting the evaluation inputs. By decoupling trust logic from raw HTTP challenges and outcome logic from raw HTTP responses, the SDK is now profoundly resilient to shifts in market hypotheses, wallet providers, and verification protocols.

## Added
* **Source-Agnostic Trust Container**: Introduced the `TrustEvidence` model. Evaluators now receive a unified evidence container holding the URL, parsed challenge, host metadata, and top-down agent hints, rather than raw HTTP parameters. This prevents lock-in to specific marketplace structures or reputation registries.
* **Agent Hints Propagation**: Extended `ExecutionContext` with a `hints` dictionary. Agents can now pass down prior knowledge, intent, or local allowlists seamlessly into the Cold Spec without requiring heavy state machines.
* **Cross-Verification in Outcomes**: Added `external_evidence` to the `OutcomeSummary` model.

## Changed
* **Outcome Verification Upgrade**: The `OutcomeMatcher` signature now natively supports accepting a `SettlementReceipt`. This enables robust "Provider-Agnostic" semantic verification—allowing agents to cross-check off-chain API responses directly against on-chain or Lightning payment proofs (e.g., verifying that a preimage matches the host's returned hash).
* **Dynamic Backward Compatibility**: The core execution engine now uses `inspect.signature` to dynamically detect the arguments expected by user-provided hooks. Evaluators and Matchers written for the v1.4.0 standard (which did not expect `TrustEvidence` or `SettlementReceipt`) will automatically fallback and function flawlessly without any code changes.
