# Release v1.5.6 — Wire-Level Protocol Purity & Interface Synchronization

This release resolves critical protocol routing issues between standard x402 and Lightning (L402/MPP) settlement layers, and fixes a latent interface bug for custom wallet adapters.

## Fixed
* **MPP Parsing Resilience**: Updated the `_parse_challenge` method to explicitly catch and process `MPP` challenges within the `WWW-Authenticate` header, preventing an erroneous fallback to EVM-based x402 logic.
* **Dynamic Scheme Resolution**: Removed the hardcoded `x402` fallback during `ParsedChallenge` instantiation. The SDK now strictly respects the `scheme` requested by the server's Base64 JSON challenge.
* **Custom Wallet Interface Bug**: Synchronized the `EVMSigner` protocol definition in `protocols.py` to match the canonical method names (`execute_lnc_evm_relay_settlement` and `execute_lnc_evm_transfer_settlement`) introduced in v1.5.2. This resolves a latent crash for developers bringing their own key adapters.

## Changed
* **Protocol Purity Enforcement**: Lightning-based settlements (L402 and MPP) now strictly transmit their proofs via the standard `Authorization` header. The SDK no longer pollutes the `PAYMENT-SIGNATURE` header or the JSON body (`paymentAuth`) during Lightning transactions, preventing routing paradoxes in proxy layers.
* **MPP Header Standardization**: The output authorization header for MPP payments has been standardized to use the `MPP <charge_id>:<preimage>` (or `MPP <preimage>`) format. This drops the legacy `Payment` prefix to align perfectly with strict server-side validation.
