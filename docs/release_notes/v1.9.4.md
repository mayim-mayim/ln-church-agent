# Release v1.9.4 — Payment Failure Observation Layer

## Summary
v1.9.4 introduces a foundational layer for treating payment failures not as binary errors, but as structured, observable friction. This allows autonomous agents to safely record, analyze, and diagnose complex friction points (like dynamic `feePayer` requirements) across the 402 ecosystem without leaking secrets.

## Added
- **`PaymentFailureRecord`**: A local Pydantic model representing a single failure observation.
- **Challenge Fingerprinting**: Deep `list`/`dict` traversal to fingerprint challenges while redacting keys like `macaroon` or `preimage`.
- **Changed Fields Detection**: Accurately detects when server requirements mutate between retries (e.g., `accepts[0].feePayer`).
- **Payload Builder**: Generates the `payment_failure_observation_report.v1` schema for future external ingestion.

## Failure Taxonomy
- Added robust primary `failure_class` categories: `retry_mismatch`, `post_settlement_proof_required`, `receipt_verification_failed`, `outcome_mismatch`.
- Added detailed `failure_subclass` mappings: `no_matching_payment_requirements`, `fee_payer_changed`, `invalid_payment_payload`.

## Safety Boundaries
- **Strict Redaction**: `macaroon`, `preimage`, and `private_key` are ignored in fingerprints and proactively `[REDACTED]` from server and client error messages.
- **Non-Verdict Model**: Records explicitly state `"not_a_verdict": true`.
- **Zero Execution Change**: This patch touches observation layers only. Existing `L402`, `x402`, and `MPP` execution paths remain 100% unaltered.

## Example
```python
from ln_church_agent.failures import build_payment_failure_record

record = build_payment_failure_record(
    endpoint="https://api.example.com/data",
    rail="x402",
    failure_subclass="no_matching_payment_requirements",
    challenge_before={"accepts": [{"feePayer": "KeyA"}]},
    challenge_after={"accepts": [{"feePayer": "KeyB"}]},
    secondary_client_used="official-x402-python"
)

# Detects ['accepts[0].feePayer']
print(record.changed_fields)

```

## Compatibility

* Fully backward compatible with v1.9.3.
* Python 3.8+ compliant.

## Future Scope

* The server endpoint `/api/agent/external/failure-observe` remains **future scope** and is not implemented yet.
* Rendering these observations in the LN Church Observatory UI is **future scope**.
* **No Automatic Submission**: This SDK version does not automatically submit failure telemetry.
* Remote MCP / Corpus MCP integration remains a separate effort.
