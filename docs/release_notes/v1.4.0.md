# Release v1.4.0 — Trust & Outcome Layer (Decide & Verify)

This major feature release transforms the SDK from a pure execution client into a highly autonomous "Decide & Verify" engine. Agents can now dynamically evaluate counterparty risk before committing funds, and semantically verify the outcome of a paid action.

## Added
* **Counterparty Trust Layer**: Introduced `TrustEvaluator` hooks and `TrustDecision` models. Agents can now intercept HTTP 402 challenges and autonomously block payments by raising a `CounterpartyTrustError` if the host or requirement fails custom security policies.
* **Outcome Verification Layer**: Introduced `OutcomeMatcher` hooks and `OutcomeSummary` models. Agents can evaluate the final HTTP 2xx response to semantically verify if the expected business outcome was achieved, storing the result directly in the `ExecutionResult`.
* **Execution Context**: Added a lightweight `ExecutionContext` (containing intent labels, session IDs, and correlation IDs) that can be passed through the `execute_detailed` execution loop to aid Evaluators and Matchers.

## Changed
* **ExecutionResult Structure**: The `ExecutionResult` model now explicitly utilizes Pydantic's `BaseModel` for robust type safety and includes an optional `outcome` field to hold the `OutcomeSummary` generated by the Matcher.
* **Client Execution Hooking**: `Payment402Client` now accepts an optional list of `trust_evaluators` on initialization. The explicit execution methods (`execute_detailed` and `execute_detailed_async`) now accept `context` and `outcome_matcher` parameters. 
* *Note: The base `execute_request` and legacy `execute_paid_action` methods remain 100% backward compatible and intentionally do not expose these advanced hooks to preserve simplicity for standard integrations.*
