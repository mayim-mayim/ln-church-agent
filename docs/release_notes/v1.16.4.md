# Release v1.16.4 — Inspect MCP SSRF and Privacy Boundary

Version 1.16.4 contains the P0-3 Inspect MCP hardening work. The Private implementation, Private-to-Public promotion identity, and Public release candidate passed independent audit before main merge. Current tag, GitHub Release, PyPI, MCP Registry, and deployment status must be verified from their respective systems rather than inferred from this source document.

## Inspect transport security

The inspect-only CLI and MCP entrypoint share a dedicated keyless transport. Inspection accepts only absolute HTTP(S) `GET` and `HEAD` requests, validates literal addresses and every DNS answer against a global-public-address policy, pins connections to validated IPs, and revalidates each redirect hop. Redirect loops, unsafe ports, private or metadata destinations, malformed locations, HTTPS downgrades, and excessive hops fail closed.

Ambient proxy variables, `NO_PROXY`, `netrc`, caller-supplied proxies, cookies, and redirected authorization credentials are not used. Requests use `Accept-Encoding: identity`; identity-encoded response bodies are read incrementally from the raw stream with a hard 1 MiB cap, while content-encoded responses are rejected before any body read. One wall-clock deadline covers DNS, connect and TLS, status and headers, body reads, and redirect processing. TLS certificate verification remains enabled for the original hostname.

## Observation and privacy boundary

Observation submission remains explicit-only and is restricted to the canonical LN Church MCP observation endpoint. The reported target is DNS-revalidated against the same public-address policy immediately before submission. The submission path does not follow redirects, performs at most one POST, closes the response after status and headers, and never reads a response body. An ambiguous timeout or network failure returns the fixed `observation_delivery_unknown` result and stops safely without replay. Only the builder's canonical allowlisted schema is accepted, and `payment_performed` remains strictly false.

Wire URLs may retain the path and query needed for the inspected request, but public MCP output, CLI JSON, observations, validators, and diagnostics retain only the initially requested URL's canonical origin. A redirect destination never replaces that public identity, and no raw path, query, or fragment is retained. Forbidden or internal URLs are replaced as a whole with `REDACTED`. Attacker-controlled public scalars are reduced to fixed values or finite allowlists. Raw invoices, amounts, payment destinations, macaroons, preimages, authorization values, cookies, private keys, signatures, proof material, receipt tokens, response bodies, resolved internal addresses, and rejected internal locations are not exposed.

## Compatibility boundary

Public HTTP(S) GET/HEAD inspection and existing 200-class, 401, 402, and 403 surface classification remain supported for AP2, ACP, APP, x402, L402, and MPP. The MCP entrypoint remains keyless and does not initialize `Payment402Client`, wallets, signers, RPC clients, payment execution, or paid retries. Canonical SVM exact remains observe-only and fail-closed at the previously established boundary.

Intentional compatibility changes reject arbitrary methods, unsafe or ambiguous URLs, ambient proxy and `netrc` behavior, caller-selected observation endpoints, raw response or exception output, and unredacted URL or payment material.
