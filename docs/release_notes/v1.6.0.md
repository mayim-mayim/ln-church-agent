# Release v1.6.0 — Internal Access Selection & Refactoring

This release focuses entirely on internal architectural improvements. It restructures the payment execution loop to support future extensions (such as discounts, subsidies, and advanced sponsor-funded access) without introducing any breaking changes or new conceptual burdens for existing users. 

The concrete usage of the 1.5.x series remains strictly intact.

## Changed (Internal Architecture)
* **Access Selection Decoupling**: The decision-making process for selecting a payment method within `LnChurchClient.draw_omikuji` (and its async counterpart) has been cleanly decoupled into separate Selector and Builder internal components. 
* **Internal Abstract Models**: Introduced private, internal-only models (`_ExecutionUnlock`, `_FundingPolicy`, `_EntitlementKind`, `_ExecutionAccessPlan`) to standardize how the SDK evaluates `direct settlement`, `faucet`, and `grant` paths.
* **Rollback Friendliness**: The core `Payment402Client` execution loop remains completely isolated from entitlement logic. Sponsor logic and legacy faucet branches are scoped securely within the LN Church reference adapter, ensuring the foundational 402 settlement runtime is unaffected.

## Compatibility Guarantee
* **Zero Breaking Changes**: There are absolutely no changes to the public API signature or the concrete vocabulary. Methods like `set_grant_token()`, `claim_faucet_if_empty()`, and the `GRANT_CREDIT` / `FAUCET_CREDIT` assets behave exactly as they did in v1.5.12.
* **Wire Protocol Consistency**: The payload structure, including how `paymentOverride` is injected during requests, remains identical down to the byte level. Existing downstream infrastructure will experience no fluctuations.

## Note for Developers
This release is fundamentally an under-the-hood optimization to prepare the SDK for upcoming agent-to-agent (A2A) economic features. No updates to your existing agent scripts or LLM tool configurations are required.
