# Release v1.8.3 — Sponsored Access Diagnostics

This release introduces local diagnostic capabilities to help agents pre-evaluate Grant tokens before execution. It clarifies Grant tokens as **Sponsored Access Entitlements** within the LN Church reference flow, separating them entirely from standard settlement rails.

## Added
* **Local Grant Diagnostics**: Introduced the `GrantDiagnostics` model and `diagnose_grant_token` helper to locally evaluate the usability of a JWS grant (expiration, audience, subject binding, scope).
* **Client Explain API**: Added `diagnose_grant()` and `explain_grant()` to `LnChurchClient`. Agents can now call `client.explain_grant()` to receive a JSON-friendly breakdown of why a grant is usable or blocked, and the recommended fallback action.
* **CLI Grant Inspector**: Added `ln-church-agent grant inspect --token <JWS> ...` to the CLI, allowing developers to locally diagnose token scopes without exposing raw token secrets to logs.

## Changed (Architecture Clarification)
* **Grant is NOT a Payment Rail**: Explicitly decoupled Grants from the settlement rail ecosystem. The diagnostics formally output `"access_path": "sponsored_grant"`, `"authorization_artifact": "scoped_grant"`, and strictly `"settlement_rail": "none"`.
* **Advisory Only**: SDK-side grant diagnostics are strictly advisory to improve UX and AI decision-making. Server-side validation remains authoritative for single-use consumption and cryptographic signature checks.

## Maintained
* **Graceful Execution Fallback**: `draw_omikuji()` seamlessly utilizes the new diagnostic logic under the hood. If a grant is found to be `expired` or `route_out_of_scope`, the client safely falls back to standard Faucet or L402/x402 execution without throwing unhandled exceptions.
* No changes to underlying L402, x402, or MPP execution logic.
