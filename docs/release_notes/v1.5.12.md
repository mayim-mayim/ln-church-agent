# Release v1.5.12 — Sponsored Access Override

This release introduces the **Sponsored Access Override**, empowering agents to bypass traditional HTTP 402 settlements by utilizing a signed, scoped `grant` token issued by a trusted sponsor. This feature serves as a foundational experiment for pre-payment distribution and sponsored execution in agent-to-agent (A2A) markets.

## Added
* **Grant Token Management**: Introduced `client.set_grant_token(token)` to store a sponsor-issued JWS grant token in the SDK client.
* **Local Policy Pre-Evaluation**: Added `has_valid_scoped_grant(target_path, method)`. The SDK now performs a lightweight, local decoding of the JWT to verify expiration (`exp`), audience (`aud`), agent identity binding (`sub`), and route/method scopes *before* transmitting the payload.
* **Asset & Scheme Expansion**: Officially added `GRANT_CREDIT` to `AssetType` and `grant` to `SchemeType`.
* **Sponsored Access Example**: Added `examples/use_grant_omikuji.py`, demonstrating a complete End-to-End flow of requesting a grant from an issuer and seamlessly consuming it.

## Changed
* **Execution Priority Routing**: Upgraded `draw_omikuji` (and its `_async` equivalent) to evaluate payment overrides natively. The SDK now evaluates the priority as follows:
  1. **Valid Grant Override** (`GRANT_CREDIT`)
  2. **Legacy Faucet Fallback** (`FAUCET_CREDIT`)
  3. **Standard Settlement** (L402 / x402)
* **Graceful Fallback**: By injecting the grant strictly as a `paymentOverride`, the core `execute_detailed` 402 retry loop remains completely untouched. If a provided grant token is expired or out-of-scope, the SDK safely ignores it and autonomously falls back to a standard L402/x402 challenge.
