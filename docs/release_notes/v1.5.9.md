# Release v1.5.9 — Evidence-Backed Session Budget Persistence

This patch reinforces the economic guardrails of autonomous agents by providing Crash/Restart Resilience for session budgets. It adheres to the SDK's core philosophy: remain strictly lightweight, completely DB-agnostic, and profoundly backward-compatible.

## Added
* **Session Budget Persistence via Evidence Sourcing**: The `PaymentEvidenceRecord` model now captures `session_spend_delta_usd`. This decouples session budget tracking from the ephemeral runtime state, treating successful 402 settlements as immutable budget events.
* **Evidence-Backed Guardrail Recovery**: Expanded the `EvidenceRepository` interface with `import_session_evidence`. If a custom repository is provided, the SDK will autonomously reconstruct the `PaymentPolicy`'s session spend limit by summing historical budget events associated with the agent's `session_id` upon execution startup.
* **Cryptographic Deduplication**: Implemented a built-in deduplication heuristic using `receipt_id` to ensure that identical settlement proofs stored in the repository (e.g., from HATEOAS recovery loops) are never double-counted against the agent's session budget.

## Changed
* **Execution Lifecycle**: Refactored `execute_detailed` and `execute_detailed_async` to perform a single, efficient, "one-shot" restore of the session budget before entering the negotiation loop.
* **Internal Rate Resolution**: Centralized USD estimation logic into a single internal helper (`_estimate_usd_value`) to ensure uniform budget calculations across execution, recording, and enforcement phases.

## Notes for Developers
* **Zero Overhead by Default**: If you are not utilizing a custom `EvidenceRepository`, this update changes absolutely nothing. The SDK remains dependency-free and continues to manage limits purely in-memory.
* **No Embedded Databases**: We have strictly avoided embedding SQLite, Redis, or local file stores. You maintain complete architectural control over where and how your agent's reasoning evidence is stored without bloating the SDK.

## Known Limitations & Considerations
For developers building custom, production-grade `EvidenceRepository` implementations (e.g., using Redis or PostgreSQL), please keep the following architectural constraints in mind:

* **Concurrency in Distributed Environments**: 
  The SDK performs a "Read-Compute-Write" cycle (it imports the session history at the start of execution, holds the budget in-memory, and exports new events upon successful settlement). If multiple agent workers process the exact same `session_id` in absolute parallel, strict budget enforcement may experience slight latency or race conditions. **Recommendation**: If strict micro-transaction caps are critical in a heavily distributed setup, consider implementing external transactional locks (e.g., Redis Mutex) at the repository level.
* **Static Exchange Rates vs. Dynamic Oracles**: 
  Currently, the internal `_estimate_usd_value` helper uses fixed, hardcoded exchange rates (e.g., 1 SATS = $0.00065). To ensure historical accuracy when rates inevitably change or when dynamic pricing oracles are introduced in the future, the SDK purposefully calculates and stores the exact `session_spend_delta_usd` *at the moment of settlement*. This prevents past budget events from artificially inflating or deflating if they are re-imported under a new rate regime.
